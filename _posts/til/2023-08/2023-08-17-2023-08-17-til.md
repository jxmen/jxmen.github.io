### React-Query 캐싱에 따른 API 설계

회사 업무를 하는 중, 피그마를 보면서 API를 설계 중이었다. 나는 단순히 한 페이지에 들어가는 내용들을 하나의 API로 설계하였다. 하지만 해당 페이지에는 서로 다른 컨텍스트로 위/아래가 나누어져 있었다. 그러자 동료가 API 설계를 미리 보고, 해당 경우는 API를 분리하는 것이 한쪽만 변경되었을 때 다른 쪽이 캐싱이 적용된다고 알려주었다.

API를 미리 만들지 않았기 때문에, 다행히 설계를 아주 빠르게 변경할 수 있었다. 만약 미리 API를 만들었다면 변경하는 데에 비용이 많이 들었거나 이 상태를 유지한 채로 갔을 것 같다.

지금 결제 기능을 구현하면서 우선 API 설계나 목업만 프론트에서 먼저 리뷰 후에 실제 API를 만드는 방식으로 하고 있는데, 이 방식이 굉장히 좋은 것 같다. (TDD의 장점을 가져온 것 같은 느낌이다) 앞으로도 이렇게 해보면 좋을 것 같다.

### API 문서화 이전에 대한 고민

현재 API 문서화에 대해 수동으로 진행하는 부분에 많은 불편함을 느끼고 있다. 지금 회사 서버는 타입도 없는 JS 일뿐만 아니라, 노션에 관련 정보를 수작업으로 만들고 있다는 것이 많이 불편했다. 또한 관련 자료를 찾아보면서 스웨거는 Open API 스펙이라는 점과, 해당 스펙을 준수하면 다른 문서화 솔루션으로 이전도 쉽다는 것을 알게 되었다.

결제 기능을 구현하면서 각종 라이브러리를 리서치해 보고 이전을 준비할 예정이다. 현재는 JSDoc으로 스웨거를 작성할 수 있는 라이브러리를 찾아 이것을 테스트해 볼 예정이다.
