git commit 메시지에 대한 내용을 정리한 글입니다.

### 커밋 메시지를 잘 써야 하는 이유

**올바른 커밋 메시지는 중요합니다.** 좋은 커밋 메시지는 협업하는 개발자가 변경된 사항에 대해 빠른 추적이 가능하게 하고, 이에 따른 빠른 버그 수정이나 follow-up이 가능하게 합니다.

대부분의 IDE에서는 git 관련 플러그인들을 제공하는데, 옆에 나오는 커밋 메시지를 통해 어느 시점에 어떻게 변경되었는지 히스토리를 파악하는 데에도 큰 도움이 됩니다.
![IDE git blame](/assets/img/posts/git/ide-git-blame.png)

- (IDE에서 내부적으로는 `git blame` 명령어를 통해 표현합니다.)

### 커밋 메시지 작성법

커밋 메시지는 한 줄 요약, 그리고 설명으로 구성되어 있습니다.

아래는 제가 개인적으로 생각하는 좋은 커밋 메시지의 예시입니다.

```shell
feat: xxx 기능 구현

- 기획 문서에 있는 내용을 토대로, ~~~을 구현했습니다.
- 앞으로 변경될만한 내용에 대비하여 ~~~ 디자인 패턴을 사용했습니다.

See Also:
```

- 첫 줄에 작업한 내용에 대해 짧게 요약한 내용을 적습니다.
- 해당 내용에 대한 부연 설명이나, 기술적은 내용은 description에 작성합니다.
- 개발할 때 참고한 문서 등이 있다면, 링크를 커밋 메시지에 같이 첨부합니다.

### 커밋 메시지 머리말

저는 주로 angular commit message 컨벤션에 따라, 머리말에 어떤 작업인지를 나타내는 단어를 사용합니다. 제가 주로 사용하는 단어들의 내용은 다음과 같습니다.

```
- feat: 기능 추가
- fix: 버그 수정
- test: 별도의 기능의 추가 없이, 테스트 코드 추가시에만 사용
- refactor: 코드 작동 내용의 변경 없이 리팩토링만 수행하였을 경우
- style: eslint, prettier 등 코드 스타일 변화만 생겼을 경우
- build: 빌드 스크립트 파일 수정 (npm)
```

- 이 부분은 속해있는 조직과 회사에 따라서 다를 수 있습니다. 예를 들어 명령조 형태의 한글만 사용하는 조직도 있습니다.
